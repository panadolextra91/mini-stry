# Mini-stry MVP Planning

You are acting as a senior staff engineer and startup technical architect.

I want you to help me design an MVP roadmap, architecture, and implementation plan.

## Product Vision

Mini-stry is a multi-tenant policy engine.

Each organization (tenant) can define its own business rules and approval workflows.

The platform itself should not contain hardcoded HR logic.

Instead, business behavior is driven by policies created by tenant owners.

The first use case is employee leave approval.

However, leave approval is only an example domain.

The actual product is the policy engine.

Future use cases may include:

- reimbursement approval
- purchase approval
- shift approval
- internal workflow approval
- any business process requiring rules and approvals

## Core Concepts

Tenant

Represents an organization.

User

Belongs to a tenant.

Role

Examples:

- employee
- manager
- hr_head
- ceo

Policy

A named set of business rules.

Policy Version

Immutable after publishing.

Only one version is active.

Request

An action submitted by a user.

Examples:

- leave request
- reimbursement request

Approval Task

Generated by policy evaluation.

Assigned to a specific approver.

Audit Log

Tracks policy publications and approval decisions.

## DSL Example

The MVP should use a text-based DSL.

Example:

name: short_leave

apply_for: all

rules:

- if: leave_days <= 2
  approve_by: manager

- if: leave_days > 2
  approve_by: ceo

The syntax does not need to be finalized.

The goal is human-readable business rules.

## Design Constraints

This is an MVP.

Do NOT build:

- payroll
- attendance
- recruitment
- performance reviews
- employee directory features beyond what is required

Focus only on:

- multi-tenancy
- policy versioning
- policy publishing
- request submission
- policy evaluation
- approval workflow generation
- audit logging
