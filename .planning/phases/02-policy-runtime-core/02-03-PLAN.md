    Create `src/modules/runtime/application/policy-runtime.ts`:
    ```ts
    import type { SchemaValidatorPort } from "../ports/schema-validator.port.js";
    import type { PolicyContent } from "../domain/policy-content.js";
    import type { EvaluationContext } from "../domain/evaluation-context.js";
    import type { EvaluationResult } from "../domain/evaluation-result.js";
    import { evaluate } from "./evaluator.js";
    import { PolicySchemaInvalidError } from "./errors.js";

    /**
     * Composes the Phase 2 runtime: validator (RUN-03) → evaluator (RUN-02) → Decision (DEC-02).
     *
     * The evaluator MUST NOT run on an invalid policy. This function enforces that contract
     * by throwing PolicySchemaInvalidError BEFORE delegating to evaluate(). RUN-03's hardest
     * invariant lives here.
     *
     * Phase 4 PolicyRuntimeService wraps this with TenantContext + persistence + audit.
     */
    export function validateAndEvaluate(
      validator: SchemaValidatorPort,
      content: unknown,
      ctx: EvaluationContext,
    ): EvaluationResult {
      const result = validator.validate(content);
      if (!result.ok) {
        throw new PolicySchemaInvalidError(result.errors);
      }
      return evaluate(content as PolicyContent, ctx);
    }
    ```

    Notes:
    - `content` is typed `unknown` because the schema validator IS the type narrower — TS cannot statically prove a `Record<string, any>` is `PolicyContent`. After a successful `validate()`, the `as PolicyContent` cast is safe by construction.
    - This is NOT a class. It is a pure function taking the validator as a parameter (D-18 spirit — port injected at call site rather than constructor since there is no state). Phase 4's `PolicyRuntimeService` will likely cache a validator instance and inject it on every request.
    - If the validator throws (compile error inside Ajv), the throw propagates — this is a programmer error, not a runtime invariant violation, and is left untouched.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Caller (Phase 4 service eventually) → validateAndEvaluate | Untrusted content (could be any JSON shape) MUST pass the validator before evaluator sees it (RUN-03). |
| validateAndEvaluate → evaluator | Trust assumption: post-validation, content is a structurally valid `PolicyContent`. The `as PolicyContent` cast is safe by construction. |
| validateAndEvaluate → PolicySchemaInvalidError consumer | The thrown error carries `ValidationError[]` (domain shape from D-22). No Ajv ErrorObject leaks. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation |
|-----------|----------|-----------|-------------|------------|
| T-02-03-01 | Elevation of Privilege | Caller bypassing the validator and calling `evaluate()` directly on un-validated content | mitigate (advisory) | Phase 2 cannot prevent direct `evaluate()` calls because both are exported. `validateAndEvaluate` is the documented entry point; Phase 4's `PolicyRuntimeService` MUST use it (call-site review, not architectural enforcement). README documents this. A v2 follow-up could lift `evaluate` out of the public surface, but for MVP keeping it exported simplifies testing (plan 02-02). |
| T-02-03-02 | Tampering (input) | Malicious JSON that passes schema validation but causes evaluator-level violations (missing field at runtime) | mitigate | EvaluationError fail-fast from plan 02-02 catches this. Test case 5 in policy-runtime.test.ts proves the EvaluationError propagates unchanged (not silently swallowed). |
| T-02-03-03 | Information Disclosure | PolicySchemaInvalidError leaking Ajv error structure to the caller | mitigate | PolicySchemaInvalidError carries `errors: readonly ValidationError[]` — the domain shape from D-22. Ajv ErrorObject was already wrapped at the adapter (plan 02-01). Composer never sees Ajv types. |
| T-02-03-04 | Repudiation | A successful evaluation produces a Decision but no trace (auditor cannot reconstruct path) | mitigate | EvaluationResult always carries `evaluationTrace: TraceEntry[]` (D-30). Tests in plan 02-02 + fixtures-e2e.test.ts assert presence and correct ordering. |
| T-02-03-05 | Spoofing (Decision tampering) | A factory returning a different Decision than promised by D-29 | mitigate | decision-factories.test.ts asserts the exact shape of every factory output, including the `@ts-expect-error` lock against accidental 4th-kind addition. |

No high-severity threats remain unmitigated. T-02-03-01 is the only advisory-level threat in Phase 2; it is documented and the v2 hardening path is clear.
</threat_model>
