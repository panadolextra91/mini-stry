# Phase 6 — UI Design Contract

> Visual and interaction contract for the Admin Portal & UI Dashboard. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Aesthetic anchor (locked, D-V2):** Minimal Linear/Vercel dark UI — deep dark surfaces, sharp typography, a single restrained accent, generous spacing, light micro-interactions. Not glassmorphism, not a data-dense console.
>
> **Theme (locked, D-V3):** Dark-mode only. No light theme this phase. HSL-variable infrastructure may make a future light theme cheap, but it is out of scope.

## Spacing Scale

8-point scale (all multiples of 4). Maps directly to Tailwind defaults (`gap-1`=4px … `gap-16`=64px).

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, inline badge padding, version-list row tightening |
| sm | 8px | Compact element spacing, button internal padding, table cell padding |
| md | 16px | Default element spacing, card internal padding, form field gaps |
| lg | 24px | Section padding, panel padding (version/lifecycle panel), card gaps |
| xl | 32px | Layout gaps between major regions, module header bottom margin |
| 2xl | 48px | Major section breaks, empty-state vertical centering padding |
| 3xl | 64px | Page-level top/bottom breathing room (the "generous spacing" of the anchor) |

**Exceptions:**
- Interactive controls (buttons, selectors, inbox row actions) have a **44px minimum hit target** even when visual height is smaller — satisfies touch/accessibility while preserving the dense-but-airy Linear feel.
- Monaco editor area is **flush** (0 internal app padding) and fills its split pane; spacing tokens apply to the surrounding chrome only.

## Typography

Four sizes, two weights (Regular 400 + Semibold 600). No other weights permitted.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Display (module/page title) | 28px | 600 | 1.2 |
| Heading (panel/section title, e.g. "Version History") | 20px | 600 | 1.2 |
| Body (default text, descriptions, table cells, form values) | 16px | 400 | 1.5 |
| Label (field labels, metadata, badges, table headers, timestamps) | 14px | 400 (600 only for active emphasis like the active-version label) | 1.5 |

**Monospace exception:** Monaco editor content, raw JSON `EvaluationContext` intake, decision-path traces, branded IDs (`policyId`, `versionId`, `taskId`) render in Geist Mono at 14px / 1.5. This is a content typeface, not a fifth UI size.

## Color

Dark-only HSL contract. 60/30/10 split. Values expressed as shadcn-style HSL token triples (set under `.dark` in `globals.css`).

| Role | Value (HSL) | Hex approx | Usage |
|------|-------------|-----------|-------|
| Dominant (60%) | `240 10% 4%` | `#0a0a0b` | App background, Monaco editor backdrop, the canvas everything sits on |
| Secondary (30%) | `240 6% 9%` | `#16161a` | Cards, sidebar/nav rail, version panel, table surfaces, dialogs, popovers — the elevated layer |
| Border / hairline | `240 5% 16%` | `#27272a` | 1px separators between panes, card borders, table row dividers (the Linear "sharp" edge) |
| Foreground (text) | `0 0% 98%` | `#fafafa` | Primary text on dominant/secondary surfaces |
| Muted foreground | `240 5% 60%` | `#94949c` | Labels, metadata, timestamps, placeholder copy |
| Accent (10%) | `217 91% 60%` | `#3b82f6` (blue) | See reserved list below |
| Destructive | `0 72% 51%` | `#dc2626` (red) | Reject / destructive-confirm only — see copywriting |

**Accent reserved for (explicit — never "all interactive elements"):**
1. The **primary CTA per screen** (e.g. Publish in the editor, Submit in Request Center, Approve in Inbox — one accent button per view).
2. The **active selection** in the left nav rail (which of the 4 modules is open).
3. The **active version** highlight in the lifecycle/version panel (D-60 — active status visually highlighted).
4. **Focus rings** on inputs/Monaco (accessibility focus state).
5. **Live-update pulse** indicating a Convex reactive change just landed (D-63 cross-user demo flow) — a brief accent flash on the updated row.

Everything else (secondary buttons, links, table chrome, version rows) stays in neutral foreground/muted/border tokens. Status semantics (request state, task state) use **muted neutral text + a small dot/badge**, with accent only for in-progress/active and destructive-red reserved for Rejected/Auto-Reject — keeping the 10% accent budget intact.

## Copywriting Contract

Primary CTAs are state/data-driven (D-57): they appear only when the lifecycle state or assignment permits. Verbs are precise; no generic "Submit" where a domain verb fits.

| Element | Copy |
|---------|------|
| Primary CTA — Editor | **Publish version** (enabled only on a valid draft; server is authoritative, D-58) |
| Primary CTA — Request Center | **Submit request** |
| Primary CTA — Inbox (per task) | **Approve** / **Reject** (shown only to the assigned approver, D-57) |
| Secondary CTA — Editor | **Save draft** (always allowed, even with advisory validation errors, D-58) |
| Secondary CTA — Version panel | **Activate** / **Roll back to this version** (shown only when lifecycle state allows, D-60) |
| Empty state — Inbox | Heading: **"No tasks waiting on you"** · Body: "Approvals assigned to the active user appear here automatically. Switch the active user to see another approver's queue." |
| Empty state — Request Center | Heading: **"No requests yet"** · Body: "Submit an EvaluationContext to run it against the active policy and watch the decision land here in real time." |
| Empty state — Policy Portal | Heading: **"No policies for this tenant"** · Body: "Create a draft to start authoring JSON rules with live schema validation." |
| Empty state — Governance Viewer | Heading: **"No audit activity"** · Body: "Published versions, rollbacks, and decisions for this tenant will be recorded here." |
| Error — Publish rejected | **"Publish blocked: the policy failed server validation."** Body: "Fix the highlighted schema errors in the editor, then publish again. (Monaco markers are advisory; the server is the source of truth.)" |
| Error — Convex query/load failure | **"Couldn't load this view."** Body: "The connection to the backend dropped. Data will refresh automatically when it reconnects — no action needed." (D-63: dropped subscription never affects correctness.) |
| Error — Invalid EvaluationContext JSON | **"That payload isn't valid JSON."** Body: "Check for a trailing comma or unquoted key, then submit again." |
| Destructive confirmation — Reject task | **"Reject this approval task?"** — "This records a Reject decision for the current stage and is captured in the audit trail. This can't be undone." Confirm button: **Reject** (destructive red). |
| Destructive confirmation — Roll back active version | **"Roll back to version {n}?"** — "This re-points the active version (D-60); historical versions stay immutable. The previously active version is preserved." Confirm button: **Roll back** (neutral, not destructive — it is non-mutating). |

**Tone:** Terse, technical, confident — matches Linear/Vercel. State labels use platform vocabulary: Draft, Published, Active, Auto-Approve, Auto-Reject, Request-Approval, Pending, Approved, Rejected.

## Interaction & Layout Contract (phase-specific)

Not in the base template but load-bearing for the checker, planner, and executor.

| Surface | Contract |
|---------|----------|
| Global shell | Persistent left nav rail (Secondary surface) listing the 4 modules; all modules always visible (no role gating, D-57). Demo Context Selector (Active Tenant + Active User) pinned top of the rail; selection drives `TenantContext {tenantId, actorId}` (D-55). Switching tenant reloads all views (D-56). |
| Admin Policy Portal (UI-01) | **Split layout** (D-60): Monaco editor = primary pane (left, ~65%), Lifecycle/Version-History panel = secondary pane (right, ~35%). Draft editable; published versions read-only; active version accent-highlighted. Live advisory Monaco markers (D-58). |
| Request Center (UI-02) | Raw JSON EvaluationContext intake (textarea/Monaco-lite) — assisted form deferred. Below/beside it: live execution-log list with decision state and step-by-step trace (mono). |
| Personal Inbox (UI-03) | List of tasks assigned to `actorId`; inline Approve/Reject per row, state-driven. |
| Governance Viewer (UI-04) | Audit timeline (chronological) + comparative policy-version list. Version-diff presentation may use a standard side-by-side/inline approach. |
| Reactivity | All four operational views use Convex reactive queries — no manual refresh (D-63). Live cross-user flow (requester submits → approver inbox updates → approver acts → requester status updates → audit timeline updates) must be visibly demonstrable. |
| Micro-interactions | Light only: 150ms ease transitions on hover/focus, brief accent pulse on reactive row updates. No heavy animation, no glassmorphism. |
